## Context

NexusBlog当前是一个前后端分离的个人博客系统，支持文章的CRUD、分类标签管理等功能。用户需要新增GitHub Markdown文档自动同步功能，实现从GitHub仓库批量拉取Markdown文件作为博客文章，减少手动发布的工作量。

本功能需要与现有系统无缝集成，不破坏原有功能，同步后的文章与原有文章统一存储和展示。

## Goals / Non-Goals

**Goals:**
- ✅ 支持多GitHub仓库配置管理，每个仓库可独立配置地址、分支、路径、令牌
- ✅ 实现手动触发的一键同步功能，拉取指定仓库的所有.md文件
- ✅ 支持Markdown文件元数据解析，自动提取标题、分类、标签、摘要等信息
- ✅ 实现去重机制，避免重复导入相同文档
- ✅ 提供同步历史记录管理，支持查看同步状态、删除同步文档
- ✅ 同步后的文章与原有文章使用相同数据结构，前台无需修改即可展示
- ✅ 无侵入式设计，不修改原有核心业务逻辑

**Non-Goals:**
- ❌ 不支持自动定时同步（后续可扩展）
- ❌ 不支持双向同步（只从GitHub拉取，不回写）
- ❌ 不支持Git子模块或嵌套仓库同步
- ❌ 不支持大型二进制文件同步

## Decisions

### 1. 架构设计
**分层架构：**
- 配置层：管理GitHub仓库配置信息，加密存储访问令牌
- 同步引擎层：负责与GitHub API交互，拉取文件列表和内容
- 解析层：解析Markdown文件内容和元数据，提取文章信息
- 业务层：处理去重逻辑、文章入库、同步记录管理
- 展示层：与原有文章系统无缝集成，统一展示

**技术选择：**
- 使用GitHub REST API v3进行仓库文件操作，支持公开和私有仓库
- 使用snakeyaml库解析Markdown YAML Front Matter元数据
- 使用文件路径+内容哈希值作为唯一标识实现去重
- 访问令牌采用AES加密存储在数据库中

### 2. 数据模型设计
新增3个数据表：
1. `github_repo_config`：存储GitHub仓库配置信息
2. `sync_history`：存储每次同步操作的记录和状态
3. `sync_article_mapping`：存储同步文章与GitHub文件的映射关系

原有`article`表无需修改，同步文章直接写入该表，通过`sync_article_mapping`关联。

### 3. API设计
遵循现有REST API规范，新增4组接口：
- 仓库配置管理接口：CRUD操作
- 同步操作接口：触发同步、查询同步状态
- 同步记录接口：查询同步历史、删除同步记录
- 同步文章管理接口：查询、删除同步的文章

### 4. 去重机制
使用双重校验：
1. 第一重：通过仓库ID+文件路径判断文件是否已经同步过
2. 第二重：通过文件内容SHA哈希值判断文件内容是否发生变化
- 若路径存在且哈希值相同：跳过
- 若路径存在但哈希值不同：更新文章内容
- 若路径不存在：创建新文章

### 5. 元数据解析规则
优先解析YAML Front Matter：
```yaml
---
title: 文章标题
date: 2024-01-01
category: 技术
tags: [Java, Spring Boot]
excerpt: 文章摘要
---
```
若无Front Matter：
- 标题使用文件名（去除.md后缀，替换连字符为空格）
- 分类默认使用配置的默认分类
- 标签默认使用配置的默认标签
- 摘要自动提取正文前200个字符

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| GitHub API限流 | 实现请求速率控制，默认1秒/请求，支持批量拉取 |
| 大文件同步超时 | 实现异步同步机制，后台任务执行，前端轮询状态 |
| 访问令牌泄露 | 令牌采用AES加密存储，数据库中不保存明文 |
| 同步失败数据不一致 | 实现事务机制，同步失败自动回滚，保持数据一致性 |
| Markdown格式不兼容 | 保留原有Markdown解析逻辑，不修改渲染规则 |
